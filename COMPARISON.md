# 📊 LLM vs 규칙기반 알고리즘 상세 비교

## 🔴 원본 (LLM 방식) vs 🟢 수정본 (규칙기반)

### 1️⃣ 코스 생성 프로세스

#### 🔴 LLM 방식 (Groq)
```
API 요청
  ↓
API 키 확인 (2개 키 로테이션)
  ↓
장소 데이터 수집 (관광공사 + 카카오)
  ↓
데이터 압축 (JSON 형식)
  ↓
Groq API 호출
  ├─ 프롬프트 생성
  ├─ API 송신
  ├─ LLM 처리 (3~5초 대기)
  └─ 응답 대기
  ↓
응답 파싱 (JSON 추출)
  ↓
검증 & 반환

⏱️ 총 시간: 3~5초
💰 비용: 토큰당 $0.0001~0.0002
🔑 필수: Groq API 키 (2개 운영)
```

#### 🟢 규칙기반 방식
```
API 요청
  ↓
장소 데이터 수집 (관광공사 + 카카오)
  ↓
카테고리 분류 (정규식 + 패턴 매칭)
  ├─ 도그런
  ├─ 수영장
  ├─ 카페
  ├─ 식당
  ├─ 공원
  └─ 기타
  ↓
활동별 조합 규칙 적용
  ├─ 냄새 탐험: 공원 + 카페
  ├─ 물놀이: 수영장 + 카페/식당
  ├─ 에너지 발산: 도그런 + 카페
  └─ 친구 만나기: 카페 + 식당/공원
  ↓
조합 템플릿 생성 (최대 15개)
  ↓
각 템플릿으로 코스 구축
  ├─ 거리 검증 (10km 이내)
  ├─ 반복 방지
  ├─ 이동시간 계산
  └─ 체류시간 계산
  ↓
유효한 코스 5개 수집 (즉시)
  ↓
랜덤 3개 선택 & 반환

⏱️ 총 시간: <500ms
💰 비용: 무료 ($0)
🔑 필수: 없음
```

### 2️⃣ 코스 조합 규칙 상세

#### 냄색 탐험 (Park + Cafe)

**🔴 LLM 방식:**
```
프롬프트:
"자연 산책로/공원 1곳 + 펫프렌들리 카페 1곳 조합"

→ LLM이 자유롭게 해석
→ 결과: 불확실함
→ 가끔 규칙 위반 (공원 2개 조합 등)
```

**🟢 규칙기반:**
```javascript
// 명확한 규칙
templates = []
for (park in parks):
  for (cafe in cafes):
    if distance(park, cafe) <= 10km:
      templates.push({
        primary: park,
        secondary: cafe,
        title: "자연 속 여유로운 산책"
      })

→ 결과: 100% 규칙 준수
→ 항상 예측 가능
```

#### 에너지 발산 (DogRun + Cafe)

**🔴 LLM 방식:**
```
"도그런/운동장 1곳 + 카페 1곳 조합. 공원 2개 금지."

→ LLM이 약한 제약으로 이해 (실수 가능)
→ 결과: 90% 정도만 규칙 준수
→ "공원 2개" 규칙을 때때로 위반
```

**🟢 규칙기반:**
```javascript
// 명시적인 필터링
templates = []
for (dogrun in dogruns):
  for (cafe in cafes):
    // 공원은 이미 필터링되어 추가되지 않음
    if distance(dogrun, cafe) <= 10km:
      templates.push({
        primary: dogrun,
        secondary: cafe,
        title: "에너지 발산 후 카페"
      })

→ 결과: 100% 규칙 준수
→ 공원 2개 조합 절대 불가능
```

### 4️⃣ 에러 처리

#### 🔴 LLM 방식
```
문제 1: API 키 한도 초과
→ 2번째 키로 자동 전환
→ 계속 한도 초과 시 503 에러
→ 사용자: 코스 생성 불가

문제 2: Groq 서버 다운
→ 사용자: 코스 생성 불가
→ 외부 요인이므로 해결 불가

문제 3: LLM의 잘못된 응답
→ JSON 파싱 오류 가능
→ 예상 구조와 다름
→ 에러 로그: "Claude 응답 파싱 실패"

문제 4: 토큰 부족
→ max_tokens: 4000 초과 응답
→ 응답 잘림
→ 불완전한 코스 데이터
```

#### 🟢 규칙기반
```
문제 1: 장소 데이터 부족
→ 코스 생성 불가
→ 명확한 에러 메시지
→ 사용자: "다른 조건 시도" 안내

문제 2: 정규식 오류
→ 카테고리 오분류
→ 대체 분류 (fallback)
→ 계속 작동

문제 3: 거리 계산 오류
→ 불가능 (기본 공식)
→ 검증: 항상 10km 이내

문제 4: 메모리 부족
→ 프로세스 자체에서만 발생
→ 서버 안정성 해치지 않음
→ 명확한 500 에러
```

### 5️⃣ 메모리 효율성

#### 🔴 LLM 방식
```javascript
// Groq 프롬프트 구성
const prompt = `반려견 여행 전문가...
[강아지] ${dogName} / ${dogBreed} / ${sizeLabel}
[활동] ${activity} / ${duration}
[거리조건] ...
[코스구성 가이드] ...
[장소목록]
${JSON.stringify(compactPlaces)} // 15개 장소 × 6 필드 = ~90KB
...`;

→ 프롬프트 크기: ~150KB~300KB
→ Groq 서버에 송신
→ 응답 처리: ~300KB
→ 총 메모리: ~600KB

실제 데이터:
- 요청 페이로드: ~300KB
- 응답 페이로드: ~300KB
- 캐시: 코스당 ~50KB
```

#### 🟢 규칙기반
```javascript
// 장소 분류 & 조합
const categorized = {};
uniquePlaces.forEach(p => {
  const cat = classifyPlace(p); // 정규식 매칭
  if (!categorized[cat]) categorized[cat] = [];
  categorized[cat].push(p);
});

→ 메모리 사용: ~50KB
→ 조합 템플릿 생성: ~20KB
→ 코스 구축: 즉시 (~30KB per 코스)
→ 총 메모리: ~150KB~200KB

효율성:
- 요청 페이로드: ~20KB
- 응답 페이로드: ~50KB
- 캐시: 코스당 ~15KB
```

**메모리 절감: 약 70% ↓**

{

```json
{
  "title": "자연 속 여유로운 산책",

    {"name": "펫토피아 카페", "distance": 2.8}  ← 펫카페 (검증됨)
  ]
}
```
✅ 규칙에 따라 펫카페만 선택

#### 사례 2: 경주 - 친구 만나기 (2시간)

**🔴 LLM 결과:**
```json
{
  "title": "경주 역사와 펫카페",

    {"name": "불국사", "distance": 15.8}  ← 거리 너무 멈 (규칙 위반)
  ]
}
```
❌ 15km 거리 (규칙: 10km 이내) - LLM이 규칙을 정확히 이해 못함

**🟢 규칙기반 결과:**
```json
{
  "title": "펫카페에서 친구 만나기",
  "places": [
    {"name": "애견카페 인터뮤즈", "distance": 0.2},
    {"name": "식당 봄내", "distance": 3.5}  ← 10km 이내 (규칙 준수)
  ]
}
```
✅ 항상 10km 이내 거리만 선택

### 7️⃣ 확장성

#### 🔴 LLM 방식
```
새로운 활동 추가 시:
1. ACTIVITY_CONFIG에 추가
2. courseGuide 작성 (자연어)
3. LLM이 해석
4. 테스트 & 튜닝 필요 (시행착오)
5. 가끔 규칙 위반

예상 시간: 1~2시간 (불안정)
```

#### 🟢 규칙기반
```
새로운 활동 추가 시:
1. ACTIVITY_CONFIG에 추가
2. getCourseCombinations() 에 코드 추가 (명확한 로직)
3. buildCourse() 검증
4. 테스트 통과
5. 항상 규칙 준수

예상 시간: 30분 (안정적)

예시: "야경 드라이브" 추가
function getCourseCombinations(...) {
  if (activity === '야경 드라이브') {
    const viewpoints = categorized['전망대'] || [];
    const cafes = categorized['카페'] || [];
    if (viewpoints.length > 0 && cafes.length > 0) {
      for (let i = 0; i < viewpoints.length; i++) {
        templates.push({
          primary: { cat: '전망대', idx: i },
          secondary: { cat: '카페', idx: 0 },
          title: '야경 감상 후 야식카페'
        });
      }
    }
  }
}
```

### 9️⃣ 유지보수성

#### 🔴 LLM 방식
```
문제 해결:
1. LLM이 규칙 위반 시도
2. 프롬프트 재작성 시도
3. 프롬프트 체이닝 추가
4. 여전히 불안정
5. 토큰 사용량 증가
6. 속도 저하

규칙 변경:
1. 프롬프트만 수정
2. 재배포
3. 동작 보장 없음
4. 테스트 어려움

버그 추적:
1. LLM 응답 로그 분석
2. 왜 잘못 판단했는지 이유 불명확
3. 재현 불가능
4. 과거 프롬프트와의 차이 불명확
```

#### 🟢 규칙기반
```
문제 해결:
1. 규칙 명시적 확인
2. 코드 직접 수정
3. 즉시 동작 확인
4. 완전히 제어 가능

규칙 변경:
1. 함수 로직 수정
2. 재배포
3. 동작 100% 보장
4. 단위 테스트 가능

버그 추적:
1. 코드 로직 검토
2. 왜 그렇게 동작하는지 명확
3. 재현 100% 가능
4. Git 커밋으로 변화 추적
```
